成都菜好吃，但仅限于刚开始的几天，餐餐都是高油高盐实在是吃不消的 。

始终对合作方的数据保持谨慎的态度，对自己的工作和成果负责，一定要对数据中不符合直觉的地方，向数据方提出质疑 。

能否识别出不符合直觉的数据问题依赖于对整个业务流程和链条的熟悉和理解！！  

在模型开发之前，数据和口径一定要确保准确，避免返工，具体包括：
1. 风险类的大口径要做到心中有数。
2. 在有类别变量时，在每个类别上的风险是否存在不符合业务直觉的地方
3. 主键是否重复，合作如何处理缺失值。
4. 合作建模的过程中，对我司与合作方的数据进行关联时，在关联过程中漏掉或者重复的数据行，弄清楚出现这些情况产生的原因。 
5. 多将风险指标与变量交叉，观察结果。

### 建立高效的工作流程链条
1. 代码中的能抽象出来的逻辑可以尽量抽象出来，不要写太长的面条式代码
2. 高内聚低耦合。
3. 确保代码可以重复运行，且满足幂等性。
4. 在分析数据的过程中，尽量将结果记录下来，便于回溯和对比。
5. 确保沟通的畅通，及时将问题暴漏出来。

### 交互决策树的开发
1. 业务逻辑相似的变量归为一类
2. 在树的同一层不要用业务逻辑不同的变量
3. 在寻找最佳分割点的时候，多尝试极端值的情况 
4. 0 1或者标签较少的变量尽量放在树根节点之下，如果每一棵树都要用到这样的变量，那么可以考虑将这些变量合并成一类，开发成一棵树。
5. 寻找最佳变量时，EM 给出的 -log(p) 可以作为参考， 但是不能盲信，最佳变量的切分还是要为业务逻辑的解释性服务。
6. 时间切片的变量在切分的时候从远到近，二级的衍生变量放在一级衍生之下，一级衍生变量放在原始变量之下，例如可以先从`近X月xxx的次数`开始分树，它的下一层节点可以选择`近x月xxx的比例` 或者 `近x月/近x月xxx的次数的趋势`。


在一周半的驻场时间中，模型的开发只占了很少的一部分，主要的时间还是在数据问题上的来回反复确认，尽快查出数据中存在的问题，就能早点完成早点回家。

以上就是在成都新网驻场进行模型开发的总结。
